# 银行 CI/CD-CMDB 系统 — 使用指引

> 面向：运维人员 / 应用负责人 / Jenkins 流水线配置者

---

## 概述

本 CMDB 是**纯配置管理数据库**，存储投产元数据，不执行任何部署动作。所有部署由 Jenkins CD 流水线通过 3 个核心 API 驱动。

---

## 一、运维管理端操作

浏览器打开 `http://{服务器地址}:8000`

### 1.3 配置变量

菜单 → **变量配置** → 新增变量

| 字段 | 说明 |
|------|------|
| 层级 | 环境/应用/集群/部署组/主机，越具体优先级越高 |
| 变量名 | 如 `DB_HOST`、`REDIS_HOST` |
| 变量值 | **明文存储**，管理端直接查看 |

**变量合并优先级**：应用全局 < 集群 < 部署组 < 单机主机

> 95% 场景只在部署组配置变量，仅特例主机做单机覆盖。

### 1.4 查看发布历史

菜单 → **发布基线**：查看每次发布的版本记录、发布时间、状态

菜单 → **审计日志**：查看所有增删改操作记录，含变更前后数据对比

## 二、Jenkins 流水线对接

### 2.1 调用流程

```
Pipeline 开始
  │
  ├─ 1. 生成 release_trace_id (UUID)
  │     POST /api/v1/cd/release-before    ← 抢占发布锁
  │
  ├─ 2. 获取配置
  │     POST /api/v1/cd/get-deploy-target ← 拿包名/路径/主机/变量/TSF配置
  │     • fixed 模式返回 instances + variables
  │     • elastic 模式返回 tsf_config + variables
  │
  ├─ 3. Jenkins 执行部署
  │     - 下载制品 biz.tar
  │     - 调蓝鲸作业传包到主机
  │     - 停服 → 清目录/覆盖 → 解压 → 启动 → 健康探测
  │     - 记录每台主机的成败
  │
  └─ 4. 回写结果
        POST /api/v1/cd/release-after      ← 写基线+释放锁
```

### 2.2 Jenkins Pipeline 示例 (Groovy)

```groovy
pipeline {
    agent any
    environment {
        CMDB_URL = 'http://cmdb.infra.bank.com'
        ENV_CODE = 'prod'
        APP_CODE = 'loan-core'
        GROUP_CODE = 'default-group'
        BUILD_NO = "${env.BUILD_NUMBER}"
        TRACE_ID = "${env.BUILD_TAG}-${UUID.randomUUID()}"
    }
    stages {
        stage('1. 发布前置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/release-before",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            release_trace_id: TRACE_ID,
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            group_code: GROUP_CODE,
                            release_no: BUILD_NO
                        ])
                    )
                }
            }
        }
        stage('2. 获取部署配置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/get-deploy-target",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            group_code: GROUP_CODE
                        ])
                    )
                    def config = readJSON text: resp.content
                    // config.data.group_type → fixed/elastic
                    // config.data.artifact_file_name → 固定包名
                    // config.data.jvm_opts → JVM参数(通用)
                    // config.data.instances → 主机列表(含单机差异化字段)
                    // config.data.tsf_config → TSF容器配置
                    // config.data.variables → 合并后环境变量
                }
            }
        }
        stage('3. 执行部署') {
            steps {
                // 遍历 config.data.instances 逐台部署
                // 单机差异化字段留空则直接用部署组默认值
            }
        }
        stage('4. 发布后置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/release-after",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            release_trace_id: TRACE_ID,
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            cluster_id: 10,
                            group_code: GROUP_CODE,
                            build_no: BUILD_NO,
                            artifact_version: "补丁${BUILD_NO}",
                            release_user: 'jenkins-cd',
                            overall_status: 'success',
                            instances: [
                                [instance_id: 1001, instance_ip: '10.20.30.1', deploy_result: 'success'],
                                [instance_id: 1002, instance_ip: '10.20.30.2', deploy_result: 'success']
                            ]
                        ])
                    )
                }
            }
        }
    }
    post {
        failure {
            httpRequest(
                url: "${CMDB_URL}/api/v1/cd/release-after",
                httpMode: 'POST',
                contentType: 'APPLICATION_JSON',
                requestBody: groovy.json.JsonOutput.toJson([
                    release_trace_id: TRACE_ID,
                    env_code: ENV_CODE,
                    app_code: APP_CODE,
                    cluster_id: 10,
                    group_code: GROUP_CODE,
                    build_no: BUILD_NO,
                    artifact_version: "失败",
                    release_user: 'jenkins-cd',
                    overall_status: 'fail'
                ])
            )
        }
    }
}
```

### 2.3 TSF 弹性容器对接

elastic 模式 `get-deploy-target` 返回 `tsf_config` 而非 `instances`：

```json
{
  "tsf_config": {
    "deploy_group_id": "tsf-group-001",
    "cpu_request": "500m",
    "cpu_limit": "2000m",
    "mem_request": "512Mi",
    "mem_limit": "2048Mi",
    "replicas": 3,
    "tsf_traffic_weight": 0,
    "update_type": 0
  }
}
```

Jenkins Pipeline 用这些字段调 TSF API 触发部署，`update_type` 决定立即/滚动更新。

## 三、场景速查

### 场景1：新应用上线

```
1. 创建环境(已有则跳过)
2. 创建系统
3. 创建应用
4. 创建集群
5. 配置部署组
6. 导入主机
7. 配置变量
8. Jenkins 触发
```
```
1. 创建环境(已有则跳过)
2. 创建应用
3. 创建集群
4. 配置部署组（虚拟机填写部署字段，TSF填写容器字段）
5. 导入主机 IP+蓝鲸五元组
6. 配置变量(DB/Redis等)
7. Jenkins 触发流水线
```

### 场景2：日常版本发布

运维端无需任何操作，配置一次永久复用。

### 场景3：回滚

回滚 = 一次新的发布。选择旧版本构建走正常流程，基线表保留历史全部版本。

### 场景4：主机维护摘除

菜单 → 主机实例 → 切换到"禁用" → 下次发布自动跳过该主机。

### 场景5：排查发布锁

```
curl http://cmdb/api/v1/cd/release-status?env_code=prod&app_code=loan-core&group_code=default-group
→ {"locked":true, "trace_id":"xxx", "ttl_seconds":3250}
```

## 四、常见问题

**Q: 部署组"新增"按钮灰了？**
A: fixed 虚拟机集群仅允许 1 条 default-group。

**Q: 变量配置了但流水线拿不到值？**
A: 检查层级优先级：主机 > 部署组 > 集群 > 应用。高层级同名变量覆盖低层级。

**Q: JVM参数配在哪儿？**
A: app_info.base_jvm_opts 是应用模板，app_deploy_group.jvm_opts 是分组级覆盖，后者优先级更高，且虚拟机/TSF通用。

**Q: 删除的数据能恢复吗？**
A: 所有删除均为逻辑删除（is_deleted=1），联系 DBA 改回 0 即可恢复。

**Q: 审计日志能删吗？**
A: 不能。sys_audit_log 永久留存，满足银行等保要求。

## 五、关键规则速记

| 规则 | 说明 |
|------|------|
| CMDB 不执行部署 | 只存配置、管状态，部署全由 Jenkins 做 |
| 固定包名 | biz.tar/app.war 文件名永不改变 |
| 版本外置 | 版本号存在 app_deploy_group_release 表 |
| 单组单发 | 一个部署组同一时间只能一场发布 |
| 自动过滤维护主机 | instance_status=0 自动跳过 |
| 三层锁防护 | DB乐观锁 → Redis SETNX → 定时兜底 |
| 变量优先级 | 主机 > 部署组 > 集群 > 应用 |
| 单机差异化 | 留空则继承部署组默认值 |
| 全部可审计 | 增删改查全程留痕 |
