# 银行 CI/CD-CMDB 系统 — 使用指引

> 面向：运维人员 / 应用负责人 / Jenkins 流水线配置者

---

## 概述

本 CMDB 是**纯配置管理数据库**，存储投产元数据，不执行任何部署动作。所有部署由 Jenkins CD 流水线通过 3 个核心 API 驱动。

### 一句话流程

```
运维配一次 → Jenkins 读配置 → Jenkins 执行部署 → 结果回写 CMDB
```

---

## 一、运维管理端操作

浏览器打开 `http://{服务器地址}:3000`（开发模式）或 Nginx 代理地址。

### 1.1 首次使用：配置四步走

**顺序不可乱**：环境 → 应用 → 集群 → 部署组

#### 第1步：创建环境
菜单 → **环境管理** → 新增环境

| 字段 | 示例 | 说明 |
|------|------|------|
| 环境编码 | `prod` | dev / test / staging / prod |
| 环境名称 | 生产环境 | |
| 权限标签 | `ops-team` | 逗号分隔，控制人员访问 |

#### 第2步：创建应用
菜单 → **应用管理** → 新增应用

| 字段 | 示例 | 说明 |
|------|------|------|
| 应用编码 | `loan-core` | 全局唯一标识 |
| 应用名称 | 贷款核心系统 | |
| 类型 | `springboot` | 选 springboot / nginx / tongweb / weblogic / tsf-service |
| 服务端口 | `8080` | 默认业务端口 |
| 负责人 | `ops-team` | |
| JVM参数模板 | `-Xms2g -Xmx4g` | 应用级默认 JVM 参数 |

#### 第3步：创建集群
菜单 → **集群管理** → 新增集群

| 字段 | 示例 | 说明 |
|------|------|------|
| 环境 | 选择 prod | |
| 应用 | 选择 loan-core | |
| 集群编码 | `prod-c1` | 集群唯一标识 |
| 集群名称 | 生产集群1 | |
| 部署模式 | `fixed` | fixed=虚拟机 / elastic=容器 |
| 状态 | 启用 | 关闭则禁止发布 |

#### 第4步：配置部署组
菜单 → **部署组管理** → 新增部署组

**必填项**：
| 字段 | 示例 | 说明 |
|------|------|------|
| 环境 | `prod` | |
| 应用 | `loan-core` | |
| 集群ID | `1` | 关联上一步创建的集群 |
| 分组编码 | `default-group` | fixed 集群仅允许此值 |
| 分组名称 | 默认分组 | |

**虚拟机部署配置**（核心）：
| 字段 | 示例 | 说明 |
|------|------|------|
| 固定包名 | `biz.tar` | 文件名永不改，不带版本号 |
| 部署策略 | `full` | full=全量(停服清目录) / incr=增量(仅覆盖) |
| 解压部署 | `是` | Y=解压到部署目录 / N=不解压直接放 |
| 部署路径 | `/data/app/loan-core` | 应用根目录 |
| 执行账号 | `appdeploy` | 将 AES 加密存储 |
| 健康检查 | `http://127.0.0.1:8080/health` | |
| JVM参数 | `-Xms2g -Xmx4g` | 部署组级 JVM（覆盖应用级） |
| 启动脚本 | `./start.sh` | |
| 停止脚本 | `./stop.sh` | |

> ⚠️ **fixed 虚拟机集群仅允许创建 1 条 default-group**，95% 场景配置一次永久复用。

### 1.2 导入主机

菜单 → **主机实例** → 新增主机

| 字段 | 示例 | 说明 |
|------|------|------|
| 部署组ID | `1` | 归属哪个部署组 |
| 内网IP | `10.20.30.1` | |
| 蓝鲸业务ID | `200` | 对接蓝鲸作业平台必填 |
| 蓝鲸主机ID | `5001` | 蓝鲸主机唯一 ID，作业首选主键 |
| 蓝鲸云区域ID | `0` | |
| 蓝鲸模块ID | `300` | |
| 标签 | `group-a` | 分批标签，如 group-a / group-b |

**批量操作**：
- 勾选多台主机 → 点击 **批量绑定部署组** → 选择目标分组
- 主机维护时切换到"禁用"状态，发布时自动跳过

### 1.3 配置变量

菜单 → **变量配置** → 新增变量

| 字段 | 说明 |
|------|------|
| 层级 | 可选 环境/应用/集群/部署组/主机，越具体优先级越高 |
| 变量名 | 如 `DB_HOST`、`REDIS_HOST` |
| 变量值 | 明文值 |
| 是否加密 | 是 = AES 加密存储，前端不显示原文 |

**变量合并优先级**：应用全局 < 集群 < 部署组 < 单机主机

> 95% 场景只在部署组配置变量，仅特例主机做单机覆盖。

### 1.4 查看发布历史

菜单 → **发布基线**：查看每次发布的版本记录、发布时间、状态

菜单 → **审计日志**：查看所有增删改操作记录，含变更前后数据对比

## 二、Jenkins 流水线对接

### 2.1 调用流程

```
Pipeline 开始
    │
    ├─ 1. 生成 release_trace_id (UUID)
    │     POST /api/v1/cd/release-before    ← 抢占发布锁
    │
    ├─ 2. 获取配置
    │     POST /api/v1/cd/get-deploy-target ← 拿包名/路径/主机/变量
    │
    ├─ 3. Jenkins 执行部署（你自己写）
    │     - 下载制品 biz.tar
    │     - 调蓝鲸作业传包到主机
    │     - 停服 → 清目录/覆盖 → 解压 → 启动 → 健康探测
    │     - 记录每台主机的成败
    │
    └─ 4. 回写结果
          POST /api/v1/cd/release-after      ← 写基线+释放锁
```

### 2.2 Jenkins Pipeline 示例 (Groovy)

```groovy
pipeline {
    agent any
    environment {
        CMDB_URL = 'http://cmdb.infra.bank.com'
        ENV_CODE = 'prod'
        APP_CODE = 'loan-core'
        GROUP_CODE = 'default-group'
        BUILD_NO = "${env.BUILD_NUMBER}"
        TRACE_ID = "${env.BUILD_TAG}-${UUID.randomUUID()}"
    }
    stages {
        stage('1. 发布前置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/release-before",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            release_trace_id: TRACE_ID,
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            group_code: GROUP_CODE,
                            release_no: BUILD_NO
                        ])
                    )
                    echo "锁抢占结果: ${resp.content}"
                }
            }
        }
        stage('2. 获取部署配置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/get-deploy-target",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            group_code: GROUP_CODE
                        ])
                    )
                    def config = readJSON text: resp.content
                    // config.data.group   → 包名/路径/脚本
                    // config.data.instances → 主机列表
                    // config.data.variables  → 环境变量
                }
            }
        }
        stage('3. 执行部署') {
            steps {
                // 遍历 config.data.instances 逐台部署
                // 记录每台主机的成败到变量中
            }
        }
        stage('4. 发布后置') {
            steps {
                script {
                    def resp = httpRequest(
                        url: "${CMDB_URL}/api/v1/cd/release-after",
                        httpMode: 'POST',
                        contentType: 'APPLICATION_JSON',
                        requestBody: groovy.json.JsonOutput.toJson([
                            release_trace_id: TRACE_ID,
                            env_code: ENV_CODE,
                            app_code: APP_CODE,
                            cluster_id: 10,
                            group_code: GROUP_CODE,
                            build_no: BUILD_NO,
                            artifact_version: "补丁${BUILD_NO}",
                            release_user: 'jenkins-cd',
                            overall_status: 'success',
                            instances: [
                                [instance_id: 1001, instance_ip: '10.20.30.1', deploy_result: 'success'],
                                [instance_id: 1002, instance_ip: '10.20.30.2', deploy_result: 'success']
                            ]
                        ])
                    )
                    echo "后置结果: ${resp.content}"
                }
            }
        }
    }
    post {
        failure {
            // 失败兜底释放锁
            httpRequest(
                url: "${CMDB_URL}/api/v1/cd/release-after",
                httpMode: 'POST',
                contentType: 'APPLICATION_JSON',
                requestBody: groovy.json.JsonOutput.toJson([
                    release_trace_id: TRACE_ID,
                    env_code: ENV_CODE,
                    app_code: APP_CODE,
                    cluster_id: 10,
                    group_code: GROUP_CODE,
                    build_no: BUILD_NO,
                    artifact_version: "失败",
                    release_user: 'jenkins-cd',
                    overall_status: 'fail'
                ])
            )
        }
    }
}
```

### 2.3 幂等保护

同一 `release_trace_id` 多次调用 `release-before` 或 `release-after` 不会重复执行。网络重试、Jenkins 重跑都安全。

## 三、场景速查

### 场景1：新应用上线

```
1. 管理员创建环境(已有则跳过)
2. 管理员创建应用               ← 1分钟
3. 管理员创建集群               ← 1分钟
4. 管理员配置部署组             ← 2分钟
5. 管理员导入主机 IP+蓝鲸五元组  ← 按主机量
6. 管理员配置变量(DB/Redis等)    ← 2分钟
7. Jenkins 触发流水线发布        ← 自动化
```

### 场景2：日常版本发布

```
1. 开发提交代码
2. Jenkins 自动构建，生成 biz.tar
3. 流水线调 release-before 加锁
4. 流水线调 get-deploy-target 拿配置
5. 流水线执行部署（停服→传包→解压→启动）
6. 流水线调 release-after 回写版本+释放锁
```
> 运维端 **无需任何操作**，配置一次永久复用。

### 场景3：回滚

```
回滚 = 一次新的发布。
1. Jenkins 选择旧版本的 artifact_version
2. 拉取对应 git commit 构建
3. 按正常发布流程走
4. release-after 写入新的基线记录
```
> 基线表保留历史全部版本，可追溯到任意时间点。

### 场景4：主机维护摘除

```
1. 菜单 → 主机实例 → 找到该主机 → 点击状态切换"禁用"
2. Jenkins 下次发布自动跳过该主机
```
> 流水线 `get-deploy-target` 接口自动过滤 `instance_status=0` 的主机。

### 场景5：排查发布锁

```
curl http://cmdb/api/v1/cd/release-status?env_code=prod&app_code=loan-core&group_code=default-group
→ {"locked":true, "trace_id":"xxx", "ttl_seconds":3250}
```
锁超时后 Redis 自动释放，APScheduler 定时任务兜底扫描僵尸锁。

## 四、常见问题

**Q: 部署组"新增部署组"按钮灰了？**
A: fixed 虚拟机集群仅允许 1 条 default-group，已有则不能再创建。

**Q: 变量配置了但流水线拿不到值？**
A: 检查变量层级。优先级：主机 > 部署组 > 集群 > 应用。高层级同名变量会覆盖低层级。

**Q: 为什么修改了部署组配置，流水线还是旧的？**
A: 变量合并有 Redis 缓存。手动修改变量/部署组后缓存自动清除，立即生效。如果仍无效，检查 Redis 是否正常。

**Q: 加密变量存哪里？**
A: `var_value` 存明文（非敏感），`encrypt_value` 存 AES-256-CBC 密文。加密密钥在 `.env` 的 `AES_SECRET_KEY`。前端默认不显示加密值，查看明文需二次授权。

**Q: 删除的数据能恢复吗？**
A: 所有删除均为逻辑删除（`is_deleted=1`），数据库记录仍在。联系 DBA 将 `is_deleted` 改回 0 即可恢复。

**Q: 审计日志能删吗？**
A: 不能。`sys_audit_log` 表设计为永久留存，无物理删除能力，满足银行等保要求。

**Q: 支持容器部署吗？**
A: 支持。集群 `deploy_mode=elastic` 时，部署组可创建多个（蓝绿/灰度），`get-deploy-target` 返回容器配置而非主机列表。

## 五、关键规则速记

| 规则 | 说明 |
|------|------|
| CMDB 不执行部署 | 只存配置、管状态，部署全由 Jenkins 做 |
| 固定包名 | `biz.tar` / `app.war` 文件名永不改变 |
| 版本外置 | 版本号存在 `app_deploy_group_release` 表，不在文件名里 |
| 单组单发 | 一个部署组同一时间只能一场发布 |
| 自动过滤维护主机 | 流水线配置接口自动跳过禁用主机 |
| 三层锁防护 | DB 乐观锁 → Redis SETNX → 定时兜底释放 |
| 变量优先级 | 主机 > 部署组 > 集群 > 应用 |
| 全部可审计 | 增删改查全程留痕，变更前后数据对比 |
