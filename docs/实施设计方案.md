# 银行 CI/CD-CMDB 实施设计方案（代码对照版）

> 本文档基于工程实际代码和数据库结构编写，与运行中代码严格一致。

---

## 0. 核心定位

- **纯配置型CMDB，无部署执行能力**：存储投产元数据、提供API查询；拉包、传包、启停等全部由 Jenkins CD 流水线执行
- **适配固定包名场景**：`biz.tar`/`app.war` 永久同名，版本外置于基线表
- **并发发布锁**：DB 乐观锁 + Redis 分布式锁双层防护
- **全操作可审计**：流水线发布全程留痕，管理端 CRUD 暂无自动切面审计

### 2.2 字段与设计文档差异说明

**⚠️ MySQL不存在设计文档声明的复合唯一索引：**
- `env_info` 缺少 `uk_env_code (env_code, is_deleted)`
- `app_deploy_group` 缺少 `uk_group_rel (env_code, app_code, cluster_id, group_code, is_deleted)`
- `cluster_instance` 缺少 `uk_group_host / uk_group_ip`
- `cicd_variable` 缺少 `uk_var_key`
- 所有表仅 `PRIMARY KEY`，无业务唯一索引

**⚠️ 字段类型差异（models.py 定义 vs 数据库）：**
- 设计文档声明 `tinyint NOT NULL DEFAULT 0`，实际数据库为 `int DEFAULT NULL`（is_deleted、lock_status 等）
- `cicd_variable` 数据库多了 `is_secret int` 和 `encrypt_value text` 两列，但 models.py 未定义

**⚠️ 排序规则：**
- 设计文档声明 `utf8mb4_unicode_ci`，实际为 `utf8mb4_0900_ai_ci`（MySQL 8.0 默认）

## 4. 发布锁机制

### 4.1 双层锁设计

**DB 乐观锁（第一层）：**
- 表 `app_deploy_group` 字段：`lock_status` / `lock_trace_id` / `lock_expire_time`
- 抢占：`UPDATE ... WHERE id=? AND lock_status=0`
- 释放：`UPDATE ... WHERE id=? AND lock_trace_id=?`（校验归属）

**Redis 分布式锁（第二层）：**
- Key: `lock:deploy:{env_code}:{app_code}:{group_code}`
- Value: `trace_id`
- 指令：`SET NX EX ttl`（兼容 Redis 5.0+）
- 释放：Lua 脚本（校验 value 后 DEL）
- Redis client: `redis.asyncio`

### 4.2 兜底机制

- 锁超时时间由入参 `timeout_minute` 控制（默认 60 分钟）
- APScheduler 兜底释放**未实现**
- 幂等校验基于 `sys_audit_log.trace_id`：同一 trace 的 PUBLISH_BEFORE / PUBLISH_AFTER 仅执行一次

## 5. 变量合并机制

### 5.1 层级优先级

```
应用级 < 集群级 < 部署组级 < 主机实例级
```

高优先级同 key 覆盖低优先级。
